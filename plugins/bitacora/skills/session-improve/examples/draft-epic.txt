Improving PROJ-2010 — "Multi-tenant data isolation"  (Epic, In Progress)
Corpus: 6 [CTX] comments · 11 free-form comments · 9 Remember hits · 17 commits · 4 PRs

--- description (proposed) ---

Goal

Isolate every tenant's data at the database level so a buggy or compromised query
cannot leak rows across tenants. Today shared tables + a `tenant_id` filter at the
ORM layer is the only boundary; we need defense-in-depth.

Scope outline

- Schema-per-tenant boundary at the Postgres level (see Assumptions for direction
  rationale)
- Migration tooling that runs DDL once and replays per-tenant
- Connection-pool changes to route to the right schema by tenant context
- Backfill of existing tenants in batches (no downtime)
- The 4 child stories referenced across the comment trail cover the four phases
  above; see them for sequencing — this epic does not duplicate their detail.

Success criteria

- Zero cross-tenant data visible in any query path, verified by the new isolation
  test suite (introduced in this epic)
- No tenant request hits the wrong schema during a one-week staging soak; p99 read
  latency stays within 5% of the pre-change baseline
- Backfill completes for all current tenants without a maintenance window
- AT-1801 (billing tenant prereq) lands before the connection-pool changes ship

Assumptions

- Schema-per-tenant chosen over row-level security — [CTX] #2 explicitly compared
  the two and landed on schema-per-tenant for read-path performance; comment #5
  from arch@ concurs. Engineer should still re-verify RLS isn't viable on a fresh
  Postgres release before locking it in.
- AT-1801 treated as a hard prerequisite (blocks the connection-pool changes), per
  comment #7. If billing decouples, the second half of this epic could ship
  independently.
- Timeline is a constraint, not a guideline (comment #11) — the 4 child stories
  span ~6 weeks of work and that window is fixed. Scope cuts come from the
  Out-of-scope list, not from extending the timeline.

Risks

- Connection-pool routing is the most under-spec'd piece (parked thread with arch@
  in comment #5); design lands or this epic stalls
- AT-1801 is on the critical path — slippage there blocks the second half
- Backfill batches may hit migration timeouts on the largest tenants; mitigation
  parked in comment #9

Out of scope

- Row-level security (rejected for read-path cost; see [CTX] #2)
- Per-tenant encryption-at-rest (separate epic, NOT this one)
- Tenant-onboarding UI changes (product epic, not platform)
